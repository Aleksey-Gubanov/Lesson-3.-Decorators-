## 🔍 Обзор проекта
Декораторы — мощный инструмент в Python для модификации поведения функций без изменения их исходного кода. В рамках данных задач реализованы механизмы перехвата аргументов, возвращаемых значений и исключений с последующей записью в файловую систему.

---

## 1️⃣ Задача 1: Базовый декоратор `logger`

**Цель:** Создать декоратор, который логирует имя функции, переданные аргументы и результат выполнения в жестко заданный файл `main.log`.

### Архитектура решения
Используется классическая структура декоратора с одним уровнем вложенности (замыкание).

**Ключевые особенности реализации:**
* **Сохранение метаданных:** Применен `@wraps` из `functools`. Это критически важно, чтобы `__name__` и `__doc__` декорированной функции не заменялись на имена внутренней функции-обертки.
* **Безопасная работа с файлами:** Использование контекстного менеджера `with open(...)`. Это гарантирует закрытие файлового дескриптора даже в случае сбоя.
* **Режим добавления (`'a'`):** Файл открывается в режиме `append`, чтобы последующие вызовы не перезаписывали предыдущие логи.

## 2️⃣ Задача 2: Параметризованный декоратор `logger(path)`

**Цель:** Доработать декоратор так, чтобы путь к лог-файлу передавался как аргумент. Добавить логирование времени вызова (timestamp) и корректную обработку возникающих исключений.

### Архитектура решения
Параметризованный декоратор требует **трех уровней вложенности** (фабрика декораторов):
1. `logger(path)` — принимает аргументы и возвращает сам декоратор.
2. `__logger(old_function)` — принимает декорируемую функцию.
3. `new_function(*args, **kwargs)` — обертка, выполняемая при вызове.

**Ключевые особенности реализации:**
* **Обработка исключений (Try/Except/Raise):** Декоратор перехватывает `Exception`, записывает тип и текст ошибки в лог, а затем **пробрасывает её дальше** (`raise`). *Правило сеньора: декоратор не должен "глотать" ошибки, если это не предусмотрено бизнес-логикой.*
* **Timestamp:** Использование `datetime.now().strftime()` для добавления человекочитаемых меток времени, что необходимо для анализа инцидентов.
* **DRY (Don't Repeat Yourself):** Строка лога формируется в памяти, а запись в файл происходит один раз в конце блока `try/except`, чтобы минимизировать операции I/O.

## 💎 Примененные Best Practices (Уровень Senior)

| Практика | Зачем это нужно |
| :--- | :--- |
| **`functools.wraps`** | Сохраняет сигнатуру, имя и docstring оригинальной функции. Необходимо для корректной работы дебаггеров, IDE и библиотек (например, FastAPI/Flask). |
| **Явный `encoding='utf-8'`** | Предотвращает `UnicodeEncodeError` при запуске на Windows, где системная кодировка по умолчанию может быть `cp1251`. |
| **Пробрасывание исключений (`raise`)** | Сохраняет оригинальный стектрейс (traceback). Позволяет внешним системам (например, Sentry) корректно отлавливать ошибки. |
| **f-strings для форматирования** | Обеспечивают максимальную читаемость кода и работают быстрее, чем `%` форматирование или `.format()`. |
| **Контекстные менеджеры (`with`)** | Предотвращают утечки файловых дескрипторов (File Descriptor Leaks) при высокой нагрузке. |
